# 3.4 — Bancos de Dados

## Conceito-base: OLTP vs OLAP

Existem **dois grandes grupos** de bancos de dados, com finalidades **opostas**. Saber diferenciar é o que mais cai na prova nessa aula.

| | **OLTP** (Online Transaction Processing) | **OLAP** (Online Analytical Processing) |
|---|---|---|
| **Finalidade** | Operação do dia a dia | Análise estratégica / BI |
| **Operação** | Escrita + leitura constantes | **Quase só leitura** |
| **Queries** | Muitas, curtas e simples | Poucas, longas e complexas |
| **Dados** | Atuais (estado mais recente) | Históricos (anos acumulados) |
| **Volume típico** | GB | **TB / PB** |
| **Latência** | Milissegundos | Segundos a minutos |
| **Estrutura** | Linhas (row-oriented) | **Colunas (columnar)** |
| **Exemplo** | "Qual o saldo do cliente X?" | "Vendas por região nos últimos 5 anos?" |
| **Serviços AWS** | **RDS, Aurora, DynamoDB** | **Redshift, Athena** |

> **Analogia:** OLTP é o **caixa da padaria** (atende rápido, muitos clientes). OLAP é o **gerente fechando o mês** (analisa tudo, leva horas).

> **Fluxo típico:** dados nascem no OLTP → ETL (AWS Glue / DMS) → carregam no OLAP → viram dashboards.

**Cenários típicos da prova:**
- "Relatório, BI, data warehouse, petabytes" → **Redshift** (OLAP)
- "Transações, checkout, cadastro" → **RDS / Aurora** (OLTP)
- "SQL serverless direto sobre S3" → **Athena** (OLAP)

## SQL vs NoSQL

Outra divisão fundamental: **como os dados são estruturados**.

| Critério | **SQL (Relacional)** | **NoSQL (Não-relacional)** |
|----------|---------------------|---------------------------|
| **Estrutura** | Tabelas com schema fixo (linhas e colunas) | Flexível (JSON, key-value, grafo, documento) |
| **Schema** | Rígido — definido antes | Dinâmico — muda a qualquer momento |
| **Linguagem** | SQL padrão | APIs próprias (cada banco tem a sua) |
| **Relacionamentos** | JOINs entre tabelas | Dados aninhados ou referências |
| **Escalabilidade** | Vertical (servidor maior) | **Horizontal** (mais servidores) |
| **Consistência** | **ACID** forte | Geralmente **eventual** (BASE) |
| **Performance** | Excelente para queries complexas | Excelente para milhões de req/s simples |
| **Volume típico** | GB a TB | TB a PB |
| **Quando usar** | Dados estruturados, transações, relatórios | Big data, IoT, mobile, alto throughput |
| **Exemplos AWS** | RDS, Aurora, Redshift | **DynamoDB**, DocumentDB, Neptune, Keyspaces |

> **Frase para a prova:** "SQL = schema rígido + JOINs + ACID. NoSQL = flexível + escala horizontal + milhões de req/s."

**Cenários típicos da prova:**
- "Schema flexível, escala massiva, latência ms" → **DynamoDB** (NoSQL)
- "Transações ACID, relacionamentos, SQL padrão" → **RDS / Aurora** (SQL)
- "Documentos JSON com MongoDB" → **DocumentDB**
- "Rede social com relacionamentos complexos" → **Neptune** (grafo)

# Bancos SQL (Relacionais)

## Amazon RDS

- **Relacional gerenciado** — AWS cuida de backup, patch, HA.
- Engines: **MySQL, PostgreSQL, MariaDB, Oracle, SQL Server, Db2, Aurora**.
- **Multi-AZ** — standby síncrono para failover (HA).
- **Read Replicas** — escalabilidade de leitura (assíncrono).

---

## Amazon Aurora

- Compatível com **MySQL e PostgreSQL** (mesmos drivers e queries).
- **5x mais rápido que MySQL · 3x mais rápido que PostgreSQL**.
- **Armazenamento auto-escalável** até 128 TB.
- **6 cópias em 3 AZs** por padrão (auto-healing de storage).
- Até **15 réplicas de leitura** (vs 5 do RDS) com lag de ~20ms.
- **Failover < 30s** (vs 60–120s do RDS Multi-AZ).
- **Backtrack** — "volta no tempo" sem precisar restore.
- **Aurora Serverless** — auto-scale e paga por uso.
- **Aurora Global Database** — replicação multi-região com lag < 1s.

### Por que o Aurora é mais rápido que o RDS?

A AWS manteve a **interface** do MySQL/PostgreSQL, mas **reescreveu o storage do zero**:

| Motivo | Explicação |
|--------|------------|
| **Storage distribuído** | 6 cópias paralelas em 3 AZs, commit com quórum (4/6) — não espera todas |
| **I/O reduzido** | Aurora envia **só redo log** ao disco (não páginas de dados) → ~6x menos I/O |
| **Réplicas eficientes** | Réplicas leem do **mesmo storage compartilhado** — não reexecutam queries |
| **Cache resiliente** | Buffer pool sobrevive a restart — não precisa "reesquentar" |
| **Failover rápido** | Réplica já está lendo o storage — só precisa virar primária |

> Para o exame, basta saber: **5x MySQL / 3x PostgreSQL · 6 cópias em 3 AZs · até 128 TB · Serverless disponível**. O "porquê" técnico cai mais em Solutions Architect.

---

## Amazon Redshift

- **Data warehouse** (OLAP) — relacional, mas voltado para análise.
- **PostgreSQL-like**, armazenamento **colunar** (otimizado para analytics).
- Petabytes em escala.
- **Redshift Serverless** disponível.

> Apesar de ser SQL, **não é OLTP** — não use para transações do dia a dia.

---

# Bancos NoSQL

## Amazon DynamoDB

- **NoSQL key-value** serverless.
- **Latência single-digit milliseconds** (< 10 ms).
- Escala para **milhões de req/s**.
- **Global Tables** — multi-região ativo-ativo.
- **DAX** — cache em memória (microssegundos).

---

## Outros Bancos NoSQL Gerenciados

| Serviço | Tipo | Caso de uso |
|---------|------|-------------|
| **DocumentDB** | Documento (MongoDB-compat.) | Documentos JSON, catálogos flexíveis |
| **Neptune** | Grafo | Redes sociais, detecção de fraude |
| **Timestream** | Séries temporais | IoT, métricas, sensores |
| **QLDB** | Ledger imutável | Auditoria, registros legais |
| **Keyspaces** | Wide-column (Cassandra-compat.) | Big data com Cassandra |
| **MemoryDB for Redis** | In-memory com durabilidade | Cache durável, alta performance |

---

# Cache In-Memory

## Amazon ElastiCache

- Cache gerenciado — **reduz carga de BD relacional**.
- Engines: **Redis** e **Memcached**.

| Característica | **Redis** | **Memcached** |
|---|---|---|
| Persistência | Sim | Não |
| Multi-AZ | Sim | Não |
| Estruturas avançadas | Listas, sets, hashes | Só strings |
| Replicação | Sim | Não |
| Caso de uso | Cache + queue + leaderboard | Cache simples e rápido |

---

# Migração

## AWS DMS (Database Migration Service)

- Migração entre bancos (**homogênea** ou **heterogênea**).
- **SCT** (Schema Conversion Tool) para conversão de engine (ex: Oracle → PostgreSQL).

---

## Pontos-Chave para o Exame

- ✅ **RDS = relacional gerenciado**; **Multi-AZ = HA**, **Read Replica = performance de leitura**.
- ✅ **Aurora** = 5x MySQL / 3x PostgreSQL, 6 cópias em 3 AZs.
- ✅ **DynamoDB = NoSQL serverless**, latência single-digit ms.
- ✅ **Redshift = data warehouse** (OLAP, não OLTP), colunar.
- ✅ **ElastiCache** = Redis (completo) ou Memcached (simples).
- ✅ **OLTP = transação · OLAP = análise**.
- ✅ **SQL = schema rígido + ACID · NoSQL = flexível + escala horizontal**.
