# 3.3 — Armazenamento

## Amazon S3 (Simple Storage Service)

- **Object storage** — arquivos em **buckets**.
- **Durabilidade 99,999999999%** (11 9's) — não perde o dado.
- **Disponibilidade 99,99%** (Standard) — acesso ao dado.
- Bucket names são **globalmente únicos** (ver detalhe abaixo).
- **Versionamento**, **lifecycle**, **replicação cross-region (CRR)**.

> ⚠️ **Não confunda:** **Durabilidade** = não perde · **Disponibilidade** = consegue acessar agora.

### Versionamento e Lifecycle

- **Versionamento:** mantém versões antigas mesmo após delete/sobrescrita. **Uma vez habilitado, só pode ser suspenso — nunca desativado**.
- **Lifecycle Policy:** regras automáticas para mover/expirar objetos. Ex: "depois de 30 dias mover para Standard-IA, depois de 90 para Glacier, deletar com 365".
- **Replicação:** CRR (Cross-Region) ou SRR (Same-Region) — copia objetos automaticamente entre buckets.

### Tipos de bucket e regras de nome

A partir do lançamento do **S3 Express One Zone** (re:Invent 2023), o S3 passou a ter **dois tipos de bucket**, com regras de nome diferentes:

| Tipo | Quando usar | Escopo do nome |
|------|-------------|----------------|
| **General purpose bucket** (padrão) | Uso geral, todas as classes (Standard, IA, Glacier...) | **Globalmente único** em toda a AWS partition (regra clássica) |
| **Directory bucket** (S3 Express One Zone) | Workloads de altíssima performance em uma única AZ | **Único apenas dentro da Zona/Região** |

- O **directory bucket** exige sufixo no nome no formato `nome-base--azid--x-s3` (ex.: `meubucket--use1-az5--x-s3`), o que escopa o nome à AZ.
- Para o exame CLF-C02, a regra que cai é a tradicional: **bucket S3 = nome globalmente único**. A exceção dos directory buckets é detalhe avançado.

### Classes de Armazenamento S3

| Classe | Caso de uso | Tempo de recuperação |
|--------|-------------|---------------------|
| **Standard** | Acesso frequente | Imediato |
| **Intelligent-Tiering** | Padrão desconhecido | Imediato |
| **Standard-IA** | Pouco acesso, rápido | Imediato |
| **One Zone-IA** | IA em uma AZ (mais barato) | Imediato |
| **Glacier Instant Retrieval** | Arquivo, acesso imediato | Imediato |
| **Glacier Flexible Retrieval** | Arquivo barato | Minutos a horas |
| **Glacier Deep Archive** | Arquivo mais barato | 12h |

## Amazon EBS (Elastic Block Store)

- **Block storage** para **EC2** — funciona como um HD/SSD.
- **AZ-específico** — ligado a uma única AZ. Para usar em outra AZ, restaure de um snapshot.
- Por padrão **conecta a uma única EC2** (Multi-Attach existe em io1/io2 mas não cai no CLF-C02).
- Tipos: **gp3/gp2** (SSD geral), **io2/io1** (SSD alto IOPS), **st1** (HDD throughput), **sc1** (HDD frio/arquivo).
- **Snapshots** — backups incrementais armazenados **no S3** (resilientes a falha de AZ — restaurável em outra AZ).

---

## Amazon EFS (Elastic File System)

- **NFS gerenciado** — compartilhado entre várias EC2.
- **Multi-AZ** por padrão.
- Linux only.

## Amazon FSx

- Sistemas de arquivos **Windows (SMB)**, **Lustre (HPC)**, **NetApp ONTAP**, **OpenZFS**.

---

## Comparação

| Recurso | S3 | EBS | EFS |
|---------|----|----|----|
| Tipo | Objeto | Bloco | Arquivo (NFS) |
| Acesso | Global via API | Uma EC2 | Várias EC2 |
| Escopo | Regional | AZ | Multi-AZ |

---

## AWS Storage Gateway

Integra on-premises com armazenamento AWS — **modelo híbrido**:

| Tipo | Cliente conecta via | Vai para | Caso de uso |
|------|---------------------|----------|-------------|
| **File Gateway** | NFS/SMB | S3 | Compartilhar arquivos com cache local |
| **Volume Gateway** | iSCSI | S3 + snapshots EBS | Block storage híbrido com backup na nuvem |
| **Tape Gateway** | iSCSI VTL | Glacier | Substitui fitas LTO físicas (mantém workflow legado) |

## AWS Snow Family

Migração **offline** de grandes volumes de dados:
| Dispositivo | Capacidade |
|-------------|------------|
| **Snowcone** | 8 TB |
| **Snowball Edge** | 80–210 TB |
| **Snowmobile** | 100 PB (caminhão!) |

## AWS Backup

- **Backup centralizado** de múltiplos serviços (EC2, EBS, RDS, DynamoDB, EFS, FSx).

## AWS DataSync

- **Transferência online** de dados on-premises ↔ AWS (também AWS ↔ AWS).
- Sincronização **incremental e contínua** pela rede.
- Diferença para Snow: **DataSync = online · Snow = offline (caminhão/avião)**.

---

## ⚠️ Armadilhas Clássicas da Prova

| Armadilha | Como evitar |
|-----------|-------------|
| **Durabilidade vs Disponibilidade** | 11 noves = não perde; 4 noves = pode acessar |
| **EBS compartilhado entre EC2** | NÃO compartilha — para isso use **EFS** |
| **One Zone-IA** | Mais barato, mas se a AZ cair, **dado pode ser perdido** — só para dados recriáveis |
| **Glacier ≠ tudo igual** | Instant (imediato) · Flexible (min/h) · Deep Archive (**12h**) |
| **S3 não é file system** | Não monta como disco — para isso, EBS (bloco) ou EFS (arquivo) |
| **Migrar PB com internet lenta** | Use **Snow Family** (offline), não DataSync nem Direct Connect |
| **Bucket name** | Globalmente único na partition — não pode repetir nem em outra região |

---

## Quando usar cada serviço (cenários típicos)

| Cenário | Serviço |
|---------|---------|
| Compartilhar arquivos entre várias EC2 Linux | **EFS** |
| Compartilhar arquivos entre várias EC2 Windows (SMB) | **FSx for Windows** |
| HPC, computação científica de alta performance | **FSx for Lustre** |
| Disco para uma única EC2 | **EBS** |
| Backup raramente acessado, custo mínimo, posso esperar 12h | **Glacier Deep Archive** |
| Acesso imprevisível, quero economizar sem complicação | **S3 Intelligent-Tiering** |
| Migrar 100 PB para AWS | **Snowmobile** |
| Migrar 50 TB para AWS | **Snowball Edge** |
| Sincronização contínua NAS on-prem ↔ S3 | **DataSync** |
| Substituir fitas LTO físicas | **Tape Gateway** |
| Backup centralizado de EC2/RDS/EFS/DynamoDB | **AWS Backup** |
| Reduzir custo automaticamente movendo objetos antigos | **S3 Lifecycle Policy** |

---

## Pontos-Chave para o Exame

- ✅ **S3 = 11 9's de durabilidade · 4 9's de disponibilidade** (Standard).
- ✅ Bucket S3 tem nome **globalmente único** (mesmo sendo serviço regional).
- ✅ **EBS** é **AZ-específico**; **EFS** é **multi-AZ Linux**; **FSx** para Windows/Lustre.
- ✅ **EBS Snapshots** ficam no **S3** — restauráveis em outra AZ.
- ✅ **Glacier Deep Archive** = mais barato, recuperação em **12h**.
- ✅ **Intelligent-Tiering** = padrão de acesso desconhecido, AWS decide.
- ✅ **One Zone-IA** = mais barato, mas dado some se a AZ cair.
- ✅ **Snowmobile** para exabytes/PB; **Snowball Edge** 80–210 TB; **Snowcone** 8 TB.
- ✅ **DataSync** online · **Snow Family** offline.
- ✅ **Versionamento** só pode ser suspenso, nunca desativado.
