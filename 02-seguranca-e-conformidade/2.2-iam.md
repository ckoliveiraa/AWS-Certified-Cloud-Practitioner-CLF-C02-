# 2.2 — IAM (Identity and Access Management)

**AWS IAM** gerencia **quem** pode fazer **o quê** em sua conta AWS. É **gratuito** e **global**.

---

## Entidades Principais

| Entidade | Descrição |
|----------|-----------|
| **Usuário** | Pessoa ou aplicação com credenciais permanentes. |
| **Grupo** | Conjunto de usuários com permissões comuns. |
| **Role (papel)** | Identidade temporária assumida por usuários/serviços. |
| **Policy** | Documento JSON que define permissões. |

---

## Tipos de Policies

- **AWS Managed** — criadas e mantidas pela AWS.
- **Customer Managed** — criadas por você, reutilizáveis.
- **Inline** — anexadas a um único usuário/grupo/role.

Exemplo de policy:
```json
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Action": "s3:GetObject",
    "Resource": "arn:aws:s3:::meu-bucket/*"
  }]
}
```

---

## Boas Práticas (caem no exame)

1. **Não use o usuário root** no dia a dia — apenas para tarefas iniciais.
2. Habilite **MFA** (especialmente no root).
3. Princípio do **menor privilégio**.
4. Use **roles** para aplicações (EC2, Lambda) — nunca grave chaves no código.
5. **Rotacione credenciais** regularmente.
6. Use **grupos** em vez de atribuir permissões por usuário.
7. **IAM Access Analyzer** — identifica recursos expostos.

---

## MFA — Multi-Factor Authentication

Algo que você **sabe** (senha) + algo que você **tem** (token/app):
- Virtual MFA (Google Authenticator, Authy)
- U2F Security Key (YubiKey)
- Hardware TOTP

---

## IAM Identity Center (antigo AWS SSO)

- Login único para múltiplas contas AWS e apps SaaS.
- Integração com **Active Directory**, **Okta**, **Azure AD**.
- Recomendado para **organizações** com várias contas.

---

## Pontos-Chave para o Exame

- ✅ IAM é **gratuito** e **global** (não tem região).
- ✅ **Root** só para tarefas iniciais — ative MFA nele.
- ✅ **Roles** para serviços AWS (EC2 → S3, por ex.).
- ✅ Policy é **JSON** com Effect, Action, Resource.
- ✅ **Identity Center** = login único multi-conta.
